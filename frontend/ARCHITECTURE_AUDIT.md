# Frontend audit: проблемы и архитектурные недостатки

## Что было не так

1. **Пустой root-компонент (`App.tsx`)**
   - Приложение не имело минимального UI-каркаса, нельзя было безопасно развивать страницы.

2. **Слабая композиция инфраструктуры**
   - API-сервисы создавались, но не было стандартного DI-слоя для React-компонентов.
   - Из-за этого каждая feature в будущем могла бы инициализировать клиентов произвольно.

3. **Непрозрачная конфигурация окружения**
   - Base URL API был разбросан по сервисному слою.
   - Не было единого модуля, который нормализует `VITE_API_BASE_URL`.

4. **Отсутствие видимого направления развития**
   - Не было технической страницы/документа, фиксирующего текущие проблемы и roadmap.

## Что улучшено

- Добавлен `app/env.ts` с нормализацией конфигурации окружения.
- Добавлен `ApiProvider` (`app/api-provider.tsx`) для централизованной инициализации API-сервисов.
- Добавлен базовый App Shell + маршрутизация на стартовую страницу.
- Добавлена стартовая страница `home-page.tsx` с:
  - перечнем найденных проблем,
  - перечнем выполненных улучшений,
  - очередью следующих архитектурных шагов.

## Рекомендуемые следующие шаги

1. Вынести все домены (`auth`, `projects`, `chats`) в полноценные feature-модули с собственными UI/API hook-композициями.
2. Добавить data cache слой (RTK Query или TanStack Query).
3. Добавить `ErrorBoundary` + telemetry (например, Sentry) на уровне app root.
4. Добавить e2e и component тесты (Playwright/Vitest) для критичных пользовательских сценариев.

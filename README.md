Главная бизнес-идея
Основа платформы формата “найти проект ↔ собрать команду ↔ откликнуться на роль ↔ работать и общаться внутри платформы”.
По сути, это социально-проектная система для коллаборации специалистов и команд (Incollab): люди создают профили, публикуют проекты, открывают позиции, получают отклики и дальше взаимодействуют в чатах/звонках. Это прямо отражено и в фронтовом позиционировании (“платформа для совместной работы над проектами”, “находите единомышленников, создавайте команды”).

Какую бизнес-задачу решает backend
Привлечение и идентификация пользователей

Регистрация/логин, refresh/logout, верификация email, reset password, OAuth-провайдеры.

То есть снижает трение входа и поддерживает безопасный жизненный цикл аккаунта.

Формирование профессиональной “витрины” пользователя

Профиль, навыки, био, контакты, аватар через presign/upload_complete.

Это нужно, чтобы участники могли оценить релевантность кандидата и строить доверие в командах.

Управление проектами и командами

CRUD проектов, личные/общие списки, инвайты, принятие инвайта, роли и permission overrides.

Это ядро бизнес-процесса “инициатор идеи → команда”.

Найм/подбор внутри проекта

Проект публикует позиции, кандидаты подают заявки, команда принимает/отклоняет.

По сути встроенный mini-ATS (applicant tracking) внутри проектной платформы.

Операционная коммуникация после мэтчинга

Чаты (direct/group/channel), управление участниками/ролями, presence, delivery, звонки через LiveKit.

Это закрывает “последнюю милю”: не только найтись, но и реально работать в одном контуре.

Архитектурная идея (почему backend устроен именно так)
Модульный монолит с CQRS-подходом: отдельные модули (auth, profiles, projects, chats), команды и запросы идут через mediator/handlers. Это удобно для роста и разделения ответственности команд разработки.

DI-контейнер Dishka для сквозной композиции модулей и зависимостей (Auth/Profile/Chat/Project providers). Это делает код расширяемым и тестируемым.

Прод-ориентированность: Redis rate limit, middleware, единый формат ошибок, Prometheus-метрики, lifecycle-инициализация брокеров/соединений. Это показывает фокус не только на фичах, но и на эксплуатации сервиса.

Коротко в одну фразу
Бизнес-цель backend-а — дать единую платформу, где специалист и проектная команда могут пройти полный цикл:
от знакомства и отбора → до совместной работы и коммуникации, без “прыжков” между разрозненными сервисами.
